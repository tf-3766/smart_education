# 知行教学云角色型 AI 助手平台设计

日期：2026-07-22  
状态：实施中  
范围：管理端、教师端、学生端、`edu-ai-service`、`edu-biz-service`、`edu-frontend`

## 1. 产品目标

AI 不是某两个固定业务的快捷入口，也不是只会回答课程资料的聊天框。最终目标是把全局 AI 助手建设为一个可持续扩展的、角色受控的业务协作入口：

- 学生能够围绕本人课程、资料、任务、考试、成绩和学习进度进行问答并获得学习辅助。
- 教师能够问答，也能用自然语言组织课程内容、教学任务、评价反馈和学情干预。
- 管理员能够问答，也能用自然语言处理其权限范围内的平台治理、教学运营和数据分析工作。
- 新业务能力通过统一能力注册和动作协议接入，不针对某一句命令写死对话流程。
- AI 只能使用当前用户真实拥有的权限和数据；关键业务结果必须经过明确的人机协作边界。

“设置某学期选课时间”“批改某次作业”仅用于说明系统应具备的通用执行能力，不是能力范围的终点。

## 2. 核心原则

1. **角色决定边界**：同一问题在学生、教师、管理员身份下可用数据和可执行动作不同。
2. **上下文决定范围**：助手感知当前页面、课程、课时和业务对象；缺少必要上下文时先澄清，不猜测 ID。
3. **查询直接执行，写入分级处理**：只读查询可直接完成；写操作按风险进入草稿、预览确认或禁止执行。
4. **业务服务是最终裁判**：AI 服务不直连 Biz 数据库。所有写入经内部契约进入 Biz，并再次鉴权、校验归属和版本。
5. **输出可验证**：资料型回答给出引用；业务动作返回对象、状态和可访问深链；失败不得伪装为成功。
6. **能力可发现、可扩展**：前端展示后端当前实际启用的能力，不用静态文案宣称尚未实现的功能。
7. **全程可审计**：记录请求人、角色、能力、业务对象、参数摘要、确认人、执行结果、模型和 traceId。

## 3. 角色能力目标

### 3.1 学生助手

- 课程资料和课时内容 RAG 问答，答案包含资料或课时引用。
- 查询本人课程、待办作业、考试安排、成绩、学习进度和本人预警。
- 根据本人已授权数据生成复习计划、知识点解释、错题分析和学习建议。
- 帮助定位页面和业务对象，可跳转到相应任务。
- 仅允许低风险的个人动作；选课、交作业、交卷等正式动作仍由学生在业务页面确认。

### 3.2 教师助手

- 具备学生侧课程知识问答能力，并能读取本人负责课程的教学上下文。
- 生成题库、作业、考试、公告、课时摘要和教学内容草稿。
- 辅助组卷、评分标准设计、提交内容分析、评语和分数建议；正式成绩由教师确认。
- 汇总课程学习情况、识别风险学生、生成干预建议和沟通草稿。
- 协助课程建设、资料组织、讨论区管理和教学运营。
- 支持单对象和批量任务，但批量执行必须展示范围、成功数、失败数和逐项结果。

### 3.3 管理员助手

- 查询平台指标、用户与教师审核状态、课程治理、选课安排、内容治理和 AI 服务状态。
- 生成平台公告、运营分析和治理建议草稿。
- 在管理员权限内提出并执行平台配置、审核和治理动作；执行前展示变更预览。
- 高风险动作必须二次确认，批量或不可逆动作还需更强确认和完整审计。
- 管理员进入课程上下文时可以进行有授权的课程分析；课程内容写入仍由 Biz 的领域权限决定。

## 4. 通用能力模型

每项 AI 能力必须注册以下元数据：

| 字段 | 含义 |
|---|---|
| `capabilityId` | 稳定能力标识，例如 `course.knowledge.qa` |
| `name` / `description` | 面向用户的名称和能力说明 |
| `roles` | 可使用的角色 |
| `mode` | `ANSWER`、`DRAFT`、`ACTION` 或 `BATCH_ACTION` |
| `riskLevel` | `READ_ONLY`、`LOW`、`MEDIUM`、`HIGH` |
| `requiredContext` | 课程、课时、作业、用户等必要上下文 |
| `confirmationPolicy` | `NONE`、`DRAFT_REVIEW`、`EXPLICIT_CONFIRM`、`STRONG_CONFIRM` |
| `deepLinkTemplate` | 完成后跳转到对应业务页面的模板 |

前端能力入口、建议问题和禁用原因都以后端返回的实际能力目录为准。

## 5. 自动流动作生命周期

所有业务自动流遵循统一状态机：

```
RECEIVED -> PLANNED -> WAITING_CONFIRMATION -> EXECUTING -> SUCCEEDED
                                      |              |-> PARTIAL_SUCCESS
                                      |              |-> FAILED
                                      |-> CANCELLED
```

一次动作至少包含：

- `actionId`、`capabilityId`、请求人和角色。
- 目标业务对象及版本，防止确认前对象已被其他人修改。
- 参数摘要和变更预览，敏感字段脱敏。
- 风险等级、确认策略和过期时间。
- 执行状态、结果资源、深链、错误码和 traceId。

模型负责理解意图、选择能力和生成候选参数；程序负责参数校验、权限、确认、幂等、并发控制和实际执行。

## 6. 风险与确认策略

| 等级 | 示例类型 | 行为 |
|---|---|---|
| `READ_ONLY` | 课程问答、查询待办、平台统计 | 可直接执行 |
| `LOW` | 生成学习计划、评语建议 | 直接返回建议，不写正式业务数据 |
| `MEDIUM` | 创建题库/作业/考试/公告草稿 | 可直接创建 `source=AI` 草稿，必须人工确认后发布 |
| `HIGH` | 正式评分、平台配置、审核、下线、批量治理 | 先预览，显式确认后执行；不可逆操作使用强确认 |

“自动完成”表示 AI 能完成理解、取数、规划、填参和调用，不表示绕过用户职责或领域安全规则。

## 7. 后端要求

- `assistantChat` 统一支持三种角色，并根据课程/页面上下文装配能力，不再用角色硬编码成两条割裂聊天路径。
- 建立能力目录，SSE 返回当前实际启用能力。
- SSE 事件至少包括 `meta`、`capability`、`tool`、`action`、`delta`、`citation`、`done`、`error`。
- 工具执行成功必须产生结构化 `action` 事件，不能只依赖模型自然语言描述。
- 大型嵌套业务对象不直接作为工具参数；采用稳定 JSON 文本解析校验，最终演进为 `draftId` 两段式提交。
- 内部写 API 透传 JWT，并在 Biz 侧重新校验身份、角色、资源归属、状态和版本。
- 后续增加持久化动作表、幂等键、确认端点和完整审计记录。

## 8. 前端要求

- 全局助手在管理端、教师端、学生端统一存在，并感知当前路由和业务对象。
- 能力条展示后端实际启用能力，不展示虚假的固定能力标签。
- 工具调用按时间线显示状态；引用以可查看来源呈现。
- `action` 事件显示结构化结果卡：草稿/执行状态、对象名称、确认要求和业务深链。
- 等待确认的高风险动作必须使用明确的确认卡片，不以聊天文字“是/否”替代正式确认协议。
- 各业务列表统一显示 `source=AI` 和草稿状态，发布/评分仍复用原业务页面及权限。
- 移动端和桌面端不遮挡主要业务操作，文本、按钮和状态不得溢出。

## 9. 非功能要求

- 权限测试覆盖跨用户、跨课程、跨角色和伪造 ID。
- 写动作具有幂等性；确认重复提交不会重复创建业务对象。
- 关键动作有超时、重试边界和可理解的失败信息。
- 对提示词注入、资料中的恶意指令、工具参数越界和隐私泄漏进行防护。
- 记录模型、提示词版本、工具输入摘要和结果，但不记录密钥或模型内部推理。
- 没有真实运行证据时不得把编译通过描述为自动流已验证。

## 10. 分阶段实施与验收

### Phase A：统一助手基础协议

- 能力目录、三角色统一路由、结构化 `action` 事件、前端工具时间线与结果卡。
- 修复复杂工具参数不稳定问题。

### Phase B：现有课程创作闭环

- 题库、作业、考试、公告全部返回 AI 草稿卡和深链。
- 对应列表统一显示来源，能够编辑并确认发布。

### Phase C：通用确认与审计执行器

- 已实现持久化动作计划、风险分级、确认/取消、幂等执行、对象版本校验和本人审计查询。
- 已用管理员“学期窗口更新”和教师“已有草稿作业发布”验证同一执行器跨角色、跨领域承载能力；它们是代表性适配器，不是对话层写死的最终能力范围。
- 后续同类能力按 `capabilityId + 参数校验 + 预览 + 领域执行适配器` 注册，无需重写助手主流程或前端确认协议。

### Phase D：教师教学协作扩展

- 作业与考试评价、批量反馈、学情分析、预警干预、课程内容维护。

### Phase E：管理员治理扩展

- 用户与教师审核、课程治理、学期运营、平台公告、内容治理和数据分析。

### Phase F：学生个性化学习闭环

- 跨本人课程的学习问答、复习规划、错题分析、进度建议和任务导航。

最终验收不是“存在几个 AI 按钮”，而是：三个角色都能在全局助手中基于真实授权数据问答；教师和管理员能通过统一协议调用其业务能力；所有写操作按风险正确确认、落库、回显、审计，且新增能力无需重写助手主流程。
